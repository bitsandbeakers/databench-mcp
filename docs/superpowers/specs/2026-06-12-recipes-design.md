# Recipe / Pipeline System — Design Spec

**Date:** 2026-06-12  
**Project:** databench-mcp  
**Status:** Approved

---

## Overview

Adds a reproducibility and sharing layer to databench-mcp. `reconstruct_recipe` walks a project's manifest, findings, and chart sidecars to generate a human-readable, standalone `recipe.py` that replays the full analysis pipeline without requiring Claude or an MCP server. `run_recipe` executes that script in-process and compares outputs against expected values using tolerance-based diffing. Two new MCP tools + two CLI subcommands expose both operations.

`EXPECTED_TOOL_COUNT` bumps from 19 → **21**.

---

## 1. Architecture

### Layer separation (unchanged pattern)

```
core/recipes.py    →  reconstruct_recipe, run_recipe, _diff_step
tools/recipes.py   →  MCP tool wrappers
server.py          →  register tools; EXPECTED_TOOL_COUNT = 21
                       CLI subcommands: recipe reconstruct / recipe run
```

### Prerequisite: chart sidecar JSON

`create_chart` currently saves only the HTML artifact. Before reconstruct can capture chart params, `core/viz.py` must save a sidecar JSON alongside each chart:

```
charts/{chart_type}_{ts}.html
charts/{chart_type}_{ts}_params.json   ← NEW
```

`_params.json` contains the full call signature:
```json
{
  "chart_type": "network_graph",
  "table": "edges",
  "columns": ["source", "target"],
  "finding_id": "f003",
  "params": {"layout": "spring", "max_nodes": 500}
}
```

This is a small addition to `_save()` in `core/viz.py` — pass `params_dict` and write it alongside the HTML.

### New workspace subdirectory

`recipes/` is already scaffolded by `workspace.py` (`_SUBDIRS`). No changes needed.

## 2. `reconstruct_recipe` Tool

### Contract

```
reconstruct_recipe(project)
→ {
    "recipe_py": str,    # absolute path to generated recipe.py
    "meta_yaml": str,    # absolute path to generated recipe_meta.yaml
    "steps": int,        # number of steps emitted
    "warning": str | None  # e.g. "recipe.py already existed and was overwritten"
  }
```

### Reconstruction algorithm

Walks project state in dependency order:

1. **Ingest steps** — for each dataset in `manifest["datasets"]`:
   - `source == "file"` → emit `ingest_file(PROJECT, "<path>", table_name="<name>")`
   - `source == "url"` → emit `ingest_url(PROJECT, "<url>", table_name="<name>")`
   - `source == "derived"` → emit `derive_table(PROJECT, "<sql>", "<name>")`
   - Then emit `profile_table(PROJECT, "<name>")`

2. **Model steps** — for each finding in `findings.yaml` (ID order, e.g. f001, f002):
   - Emit `run_model(PROJECT, "<table>", "<method>", target=..., features=[...], params={...})`

3. **Chart steps** — for each `{ts}_params.json` sidecar in `charts/` (sorted by ts):
   - Emit `create_chart(PROJECT, "<chart_type>", "<table>", columns=[...], ...)`
   - Skip with warning if referenced `finding_id` no longer exists

Ordering is deterministic: `assert_profiled` enforces that ingest/derive always precedes any model or chart step, and findings carry the table they were run against.

### Generated `recipe.py` shape

```python
# Generated by databench-mcp reconstruct_recipe — do not edit manually.
# Run standalone: pip install databench-mcp && python recipe.py
from databench_mcp.core.ingestion import ingest_file
from databench_mcp.core.eda import derive_table
from databench_mcp.core.profiler import profile_table
from databench_mcp.core.modeling import run_model
from databench_mcp.core.viz import create_chart

PROJECT = "my-project"

def run():
    ingest_file(PROJECT, "/abs/path/data.csv", table_name="providers")
    derive_table(PROJECT, "SELECT src, tgt FROM providers", "edges")
    profile_table(PROJECT, "edges")
    run_model(PROJECT, "edges", "network_communities",
              params={"source_col": "src", "target_col": "tgt", "seed": 42})
    create_chart(PROJECT, "network_graph", "edges",
                 columns=["src", "tgt"],
                 params={"layout": "spring"})

if __name__ == "__main__":
    run()
```

Importable as a library AND runnable as a script. The only dependency is `pip install databench-mcp`.

### Generated `recipe_meta.yaml` shape

```yaml
project: my-project
reconstructed_at: "2026-06-12T13:00:00Z"
tolerances:
  numeric_rtol: 1.0e-5
  numeric_atol: 1.0e-8
steps:
  - kind: ingest_file
    params: {source: "/abs/path/data.csv", table_name: "providers"}
    expected: null
  - kind: derive_table
    params: {sql: "SELECT src, tgt FROM providers", table_name: "edges"}
    expected: {rows: 450, columns: 2}
  - kind: profile_table
    params: {table: "providers"}
    expected: null
  - kind: run_model
    params:
      table: edges
      method: network_communities
      params: {source_col: src, target_col: tgt, seed: 42}
    expected:
      metrics:
        num_communities: 5
        modularity: 0.412
  - kind: create_chart
    params:
      chart_type: network_graph
      table: edges
      columns: [src, tgt]
      params: {layout: spring}
    expected: null   # layout is non-deterministic; chart diff skipped
last_run: null       # populated after first run_recipe call
```

### Error conditions

| Condition | Error |
|-----------|-------|
| No datasets in manifest | `ValueError: nothing to reconstruct — ingest data first` |
| Chart sidecar references missing `finding_id` | warning in return value; step skipped |
| `recipe.py` already exists | overwritten; warning returned |

## 3. `run_recipe` Tool

### Contract

```
run_recipe(project, diff_mode=True)
→ {
    "status": "clean" | "changed" | "error",
    "steps_run": int,
    "changes": [           # empty if clean
      {
        "step": int,
        "kind": str,
        "field": str,
        "expected": Any,
        "actual": Any,
        "delta": float | None
      }
    ],
    "finding_ids": [str],  # finding IDs created during this run
    "error": str | None
  }
```

### Execution flow

1. Snapshot existing finding IDs from `findings.yaml`
2. Load and execute `recipe.py`'s `run()` via `importlib.util` in-process
3. Detect new findings: diff `findings.yaml` before/after; new entries = this run's output
4. If `diff_mode=False`: update `recipe_meta.yaml` unconditionally and return `"clean"`
5. Compare new findings' metrics against `steps[*].expected` by step position (not finding_id)
6. If all diffs pass → update `recipe_meta.yaml` (expected values + `last_run`), return `"clean"`
7. If any diff fails → return `"changed"` with per-step details; do NOT update expected

### Diff rule

- **Numeric float:** `abs(new - old) <= atol + rtol * abs(old)` → pass
- **Integer / count:** exact match required (e.g. `num_communities`, `rows`)
- **Non-numeric strings:** exact match
- **`expected: null`:** step always passes (no comparison)
- **Charts:** never diffed (non-deterministic layout); `expected` is always null for `create_chart` steps

### Finding ID management

Old findings accumulate normally in `findings.yaml` — `run_recipe` does not delete them. `last_run.finding_ids` in `recipe_meta.yaml` is the pointer to what the last recipe execution produced. This allows comparing last-run vs. current-run without polluting the interactive findings log.

### CLI commands

```bash
databench-mcp recipe reconstruct <project>
databench-mcp recipe run <project> [--no-diff]
```

Both added to the `argparse` block in `server.py:main()`.

## 5. Testing Strategy

```
tests/
  test_core_recipes.py            # unit: reconstruct output, run diff=clean, run diff=changed
  tools/
    test_recipes_integration.py   # full pipeline → reconstruct → run → confirm status=clean
```

### `test_core_recipes.py`

- `test_reconstruct_generates_valid_python` — reconstruct on a project with ingest + derive + model + chart; verify `recipe.py` is importable and `recipe_meta.yaml` is valid YAML
- `test_reconstruct_empty_project_raises` — no datasets → `ValueError`
- `test_run_recipe_clean` — run_recipe on a freshly reconstructed project; status is `"clean"`
- `test_run_recipe_changed` — manually tweak `expected.metrics` in recipe_meta.yaml to a wrong value; status is `"changed"` with correct change entry
- `test_run_recipe_no_diff` — `diff_mode=False` always returns `"clean"` and updates expected

### `test_recipes_integration.py`

1. Ingest synthetic CSV → `derive_table` → `profile_table` → `run_model` → `create_chart`
2. `reconstruct_recipe` → verify `recipe.py` and `recipe_meta.yaml` written
3. `run_recipe` → verify `status == "clean"` and `finding_ids` is non-empty
4. Run again → verify idempotent (`status == "clean"`, expected values unchanged)

## 6. Extensibility

To add per-step tolerance overrides later:
- Add `tolerances:` key under each step in `recipe_meta.yaml` (overrides top-level)
- `_diff_step` checks step-level tolerances first, falls back to project-level

To add directed `run_recipe` (only re-run from a specific step):
- Add `from_step: int` param to `run_recipe`; skip steps before it

No changes to `server.py` or tool contracts for either extension.
